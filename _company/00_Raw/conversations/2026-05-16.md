## 스파인 2D 스테이그벌레 애니메이션 분류 구조 가이드

게임 성능과 리소스 효율성을 최우선으로 고려하여 스테이그벌레 캐릭터의 동작 애니메이션을 분류하는 것은 복잡한 움직임을 체계적으로 관리하고 리소스를 절약하는 데 필수적입니다. 주요 카테고리와 구체적인 예시는 다음과 같습니다.

**1. 기본 이동 (Basic Movement)**
캐릭터의 기본적인 위치 변화와 이동에 초점을 맞춥니다. 이는 가장 빈번하게 사용되며, 반복적인 움직임이므로 효율적인 애니메이션 제작이 중요합니다.
*   **예시:** 일직선 이동(좌/우/상/하), 부드러운 기어 이동(Crawling), 멈춤(Idle).

**2. 동작 (Actions)**
특정 상태 변화나 목적을 달성하기 위한 연속적인 움직임을 포함합니다. 이는 캐릭터의 '상태'를 정의하는 애니메이션입니다.
*   **예시:** 먹이 섭취(Eating Sequence), 휴식/잠자기(Sleeping Pose), 포식자 회피 자세(Fleeing).

**3. 상호작용 (Interactions)**
다른 오브젝트와의 물리적 또는 논리적 접촉에 반응하는 움직임을 정의합니다. 이는 게임의 인터랙션 피드백을 제공합니다.
*   **예시:** 벽 기어오르기(Climbing), 장애물 회피 충돌(Collision Avoidance), 물체 밀기/당기기(Push/Pull).

**4. 특수 효과 (Special Effects)**
기본 이동이나 동작에 부가되어 시각적인 재미를 더하는 비주요 애니메이션입니다. 이는 리소스를 적게 사용하면서도 캐릭터의 개성을 살립니다.
*   **예시:** 꼬리 흔들림(Tail Wiggle), 빛 발산(Glow Effect), 감정 표현(Reaction/Surprise).

이러한 분류를 통해 각 애니메이션을 독립적인 클립으로 분리하여 Spine에서 효율적으로 관리할 수 있으며, 필요한 시점에만 해당 리소스를 호출하여 게임 성능 저하를 최소화할 수 있습니다.

## ⚙️ Spine 2D와 게임 엔진 연동을 위한 성능 최적화 다이어그램 브리프 (텍스트 기반)

이 문서는 Spine에서 제작된 캐릭터 애니메이션 데이터를 Unity 또는 Godot와 같은 게임 엔진에 효율적으로 통합하고, 특히 성능(CPU/GPU 부하)을 최소화하기 위한 뼈대 계층(Hierarchy) 및 블렌딩 트리(Blending Tree)의 활용 방안을 설명합니다.

### 1. 핵심 원칙: 계층 구조 (Skeleton Hierarchy)의 역할

**목표:** 불필요한 계산을 줄이고, 애니메이션 업데이트를 효율적으로 수행하기 위해 뼈대(Bone) 구조를 논리적인 계층으로 설계해야 합니다.

*   **최상위 레벨 (Root/Controller):** 캐릭터 전체의 움직임이나 상태 변화를 제어하는 최상위 컨트롤러 본(Bone)을 설정합니다. (예: `Root`, `Body_Controller`)
    *   **역할:** 이 뼈대의 변형은 다른 모든 하위 뼈대에 영향을 미치며, 게임 엔진에서 캐릭터의 전체 위치나 회전 변화를 직접 제어하는 데 사용됩니다.
*   **중간 레벨 (Body/Limb Groups):** 신체 부위별 그룹을 정의합니다. (예: `Torso`, `Upper_Arm`, `Lower_Leg`)
    *   **역할:** 이 레벨의 변형은 해당 부위의 애니메이션(팔, 다리)이 자연스럽게 연결되도록 보장하며, 물리적 상호작용 시 관절의 움직임을 모델링합니다.
*   **최하위 레벨 (Joint/Endpoint):** 실제 메시의 앵커 포인트나 세부적인 관절을 정의합니다. (예: `Shoulder_Joint`, `Elbow_Joint`)
    *   **역할:** 이 레벨이 애니메이션의 최종 시각적 디테일을 결정하며, 게임 엔진에서 물리 충돌이나 메시 변형에 직접 매핑됩니다.

**성능 최적화 포인트:**
1.  **계층 단순화:** 꼭 필요한 뼈대만 유지하여 메모리 사용량을 줄입니다.
2.  **관절 최소화:** 애니메이션에 영향을 미치지 않는 불필요한 뼈대의 중첩을 피합니다.

### 2. 동적 움직임 처리: 블렌딩 트리 (Blending Tree)의 활용

**목표:** 여러 애니메이션 상태(걷기, 뛰기, 점프 등) 사이를 부드럽게 전환하며, 복잡한 움직임을 효율적으로 관리합니다.

*   **State Machine 연결:** 게임 엔진의 상태 머신(State Machine)이 결정한 현재 상태에 따라 Spine에서 준비된 애니메이션 클립을 선택하고 블렌딩합니다.
    *   예시: `Idle` 상태 $\rightarrow$ `Walk` 상태로 전환 시, 두 상태의 중간 프레임을 수학적으로 혼합하여 부드러운 보간(Interpolation)을 수행합니다.
*   **Layered Blending (레이어드 블렌딩):** 복잡한 캐릭터 움직임에서 신체 각 부분을 독립적으로 제어할 수 있게 합니다.
    *   예시: `상체(Torso)`는 걷기 애니메이션으로, `팔(Arms)`은 별도의 동작(예: 무기 휘두르기) 애니메이션을 블렌딩하여 적용합니다. 이는 전체 캐릭터의 움직임이 아닌, **관절 단위**의 미세한 동적 변화를 표현할 때 필수적입니다.

**게임 엔진 연동 시 흐름:**
1.  **게임 로직 (Unity/Godot):** 플레이어 입력이나 환경 반응에 따라 현재 상태(State)를 결정합니다.
2.  **데이터 전송:** 결정된 State ID와 타겟 Transition을 Spine 런타임으로 보냅니다.
3.  **Spine 처리:** Spine은 해당 State에 맞는 애니메이션 클립들을 로드하고, 정의된 블렌딩 규칙에 따라 프레임을 계산하여 결과(Pose)를 반환합니다.
4.  **엔진 적용:** 엔진은 Spine이 반환한 최종 뼈대 위치 데이터를 받아 실제 메시(Mesh)의 버텍스(Vertex) 위치를 업데이트하여 화면에 렌더링합니다.

## 🌿 사슴벌레 캐릭터 동작 애니메이션 분류 가이드 (Spine 활용)

사슴벌레와 같은 곤충형 캐릭터는 움직임이 비교적 단순하지만, 환경과의 상호작용을 표현하기 위해 아래의 분류 구조를 사용하면 효율적으로 리소스를 관리하고 다양한 인게임 상황을 커버할 수 있습니다.

### 1. 기본 이동 (Basic Movement) - 핵심 구동부

**목표:** 캐릭터가 공간 내에서 움직이는 가장 기본적인 상태를 정의합니다.
*   **이동 유형:**
    *   **직선 이동:** 앞/뒤/좌/우로 일정한 속도로 움직임 (기본 이동, 탐색).
    *   **Crawling (기어 이동):** 사슴벌레 특유의 느리고 굴러가는 듯한 움직임. 이는 일반적인 `Walk`보다 더 부드럽고 리듬감 있게 설계합니다.
    *   **Idle (정지):** 완전히 멈춰 있을 때의 포즈.
*   **활용:** 이 애니메이션들은 가장 높은 빈도로 사용되므로, **가장 최적화된(낮은 폴리곤 수의) 베이스 애니메이션**으로 제작해야 합니다.

### 2. 동작 (Actions) - 상호작용 및 상태 변화

**목표:** 특정 목적을 달성하거나 환경과 상호작용할 때 발생하는 동적인 움직임을 정의합니다.
*   **상태 전환 동작:**
    *   **Stop/Start Transition:** 정지에서 이동으로, 이동에서 멈춤으로 넘어갈 때의 부드러운 연결 애니메이션. (블렌딩에 필수)
    *   **Resting Pose:** 잠자기, 휴식 등의 상태 변화 시의 고정된 포즈.
*   **상호작용 동작:**
    *   **Climb/Cling:** 벽이나 표면을 기어오르거나 달라붙는 움직임. (특수 뼈대 사용)
    *   **Attack/Sting:** 상대에게 접근하거나 독침을 사용하는 순간의 짧고 역동적인 동작.

### 3. 특수 효과 (Special Effects) - 개성 및 디테일

**목표:** 캐릭터의 개성을 살리거나, 기본 움직임에 추가적인 시각적 재미를 부여하는 애니메이션입니다.
*   **신체 디테일:**
    *   **Wiggle/Vibration:** 이동 중 몸체나 다리가 미세하게 떨리는 효과 (물리적 충격을 표현).
    *   **Tail Flutter:** 꼬리를 흔들거나 움직이는 애니메이션. (이동과 독립적으로 블렌딩 가능)
    *   **Glow Effect:** 특정 상태(예: 위험 감지, 포식자 등장)에서 발생하는 빛 효과의 시각적 변화.

### 요약 및 권장 사항

사슴벌레 캐릭터에서는 **1번(기본 이동)**을 중심으로 애니메이션을 구성하고, **3번(특수 효과)**을 보조적으로 활용하여 움직임에 생동감을 더하는 것이 성능과 디자인 목표를 모두 달성하는 최적의 방법입니다. 모든 애니메이션은 Spine에서 **최소한의 뼈대**만 사용하여 제작해야 게임 엔진으로의 연동 시 오버헤드를 최소화할 수 있습니다.

3.  **실시간 동기화 및 알림 (Real-time Sync & Notification):**

4.  **통합 분석 대시보드 (Analytics Dashboard):**
    *   어떤 채널에서 메뉴 업데이트가 성공했는지, 어떤 채널에 대한 고객 반응(리뷰 등)이 있는지를 추적하여 마케팅 인사이트 제공.

## 1. 핵심 가치 제안 (Core Value Proposition - CVP)

**[Pain Point 정의]**
레스토랑 오너들이 여러 플랫폼(구글, Yelp, 자체 채널 등)에 메뉴를 수동으로 복사/붙여넣기 하면서 발생하는 **정보 불일치, 시간 낭비, 그리고 고객 경험 저하**의 고통을 해결합니다.

**[핵심 가치 제안 (CVP)]**
> **"하나의 시스템에서 모든 플랫폼의 메뉴를 완벽하게 동기화하세요. 이제 메뉴 업데이트는 단 한 번의 클릭으로 끝납니다."**

**[보조 가치 (Benefits)]**
*   **완벽한 일관성:** 모든 온라인 채널(웹사이트, 지도, 예약 플랫폼)에 실시간으로 동일한 메뉴가 반영되어 고객 혼란을 원천 차단합니다.
*   **시간 절약:** 수동 복사/붙여넣기 및 오류 수정에 드는 시간을 획기적으로 줄여, 오너님이 핵심 비즈니스에 집중할 수 있게 합니다.
*   **운영 효율성:** 메뉴 변경 시 발생하는 실수와 운영상의 스트레스를 제거하여, 매일 더 효율적으로 레스토랑을 운영하게 돕습니다.

## ⚙️ 시스템 아키텍처 설계

### 1. 데이터베이스 스키마 설계 (MongoDB Conceptual Schema)

데이터의 일관성과 관계를 보장하기 위해 핵심 엔티티와 관계를 정의합니다. MongoDB의 유연성을 활용하되, 메뉴 동기화에 필요한 관계 필드를 명시합니다.

**Collection 1: Restaurants (레스토랑 정보)**
*   `_id`: ObjectId
*   `name`: String (레스토랑 이름)
*   `ownerId`: ObjectId (사용자 ID 참조)
*   `settings`: Object (디자인 설정, 기본 통화 등 - 보안을 위해 환경 변수와 연동 필요)

**Collection 2: Menus (메뉴 정보)**
*   `_id`: ObjectId
*   `restaurantId`: ObjectId (Restaurant 참조)
*   `menuName`: String (예: '오늘의 특선', '런치 메뉴')
*   `status`: String (Draft, Published, Archived)
*   `content`: Array of MenuItems (실제 메뉴 항목 배열)
*   `updatedAt`: Date

**Collection 3:MenuItems (개별 메뉴 항목)**
*   `_id`: ObjectId
*   `menuId`: ObjectId (Menu 참조)
*   `itemName`: String (메뉴 이름)
*   `description`: String (상세 설명)
*   `price`: Number (가격)
*   `category`: String (카테고리, 예: Appetizer, Main Course)
*   `imageUrl`: String (사진 URL)
*   `platformSyncStatus`: Object (각 플랫폼별 동기화 상태 추적 - 향후 확장 대비)

**Collection 4: PlatformSyncLogs (플랫폼 동기화 기록)**
*   `_id`: ObjectId
*   `menuId`: ObjectId
*   `platform`: String (예: GoogleMaps, Yelp, WebsiteCMS)
*   `syncDate`: Date
*   `status`: String (Success, Failed, PendingReview)
*   `errorDetails`: String (실패 시 에러 로그 기록)

### 2. API 구조 설계 (Next.js API Routes / Node.js Backend Logic)

API는 **RESTful** 원칙을 따르며, 메뉴 관리의 핵심인 CRUD와 동기화 로직에 집중합니다.

| Endpoint | Method | 설명 | 접근 권한 | 주요 기능 |
| :--- | :--- | :--- | :--- | :--- |
| `/api/restaurants` | `GET` | 특정 레스토랑 정보 조회 | Owner Only | 레스토랑 목록 및 상세 정보 조회 |
| `/api/restaurants/:id` | `GET`/`PUT` | 레스토랑 데이터 관리 | Owner Only | 레스토랑 설정 수정 |
| `/api/menus` | `POST` | 새로운 메뉴 생성 | Owner Only | 새 메뉴(Menu) 등록 |
| `/api/menus/:menuId` | `GET`/`PUT` | 특정 메뉴 내용 조회/수정 | Owner Only | 메뉴 콘텐츠 상세 편집 |
| `/api/sync/trigger` | `POST` | **동기화 트리거** | Owner Only | 모든 플랫폼으로 메뉴 업데이트 요청 (핵심 로직) |
| `/api/sync/logs` | `GET` | 동기화 로그 조회 | Owner Only | 과거 동기화 성공/실패 기록 확인 |

### 3. 프론트엔드 컴포넌트 아키텍처 (React)

디자인 브리프에서 정의된 **신뢰감**과 **식욕 자극**을 시각적으로 구현하며, 기능별로 모듈화합니다. 상태 관리는 Recoil 또는 Redux/Zustand를 고려하여 중앙 집중화합니다.

**A. Layout & Shell Components (뼈대)**
*   `Layout`: 전체 페이지 레이아웃 (Header, Sidebar) 관리.
*   `Sidebar`: 네비게이션 메뉴 및 설정 접근.
*   `DashboardView`: 메인 대시보드 컴포넌트 (통계 카드).

**B. Core Feature Components (핵심 기능)**
1.  **`MenuEditor`**: 메뉴 목록 및 개별 메뉴 항목을 편집하는 핵심 폼 컴포넌트. (가장 많은 상호작용 발생)
2.  **`PlatformSyncManager`**: 각 외부 플랫폼(Google, Yelp 등)에 대한 동기화 설정 및 상태를 시각적으로 보여주는 관리 컴포넌트.
3.  **`DataTable`**: 메뉴 목록을 테이블 형태로 표시하며 필터링/정렬 기능을 제공.

**C. Service Components (데이터 처리)**
*   `ApiService`: 백엔드 API 통신을 전담하는 훅(Hook) 모듈. (보안 및 에러 처리를 중앙화)
*   `SyncService`: 실제 `POST /api/sync/trigger` 호출 및 로깅을 담당하는 비즈니스 로직 레이어.
